| 日期 | 工具 | 错误现象 | 根因 | 解决方案 | 预防措施 |

| 6.3 | Python | `Python was not found` — 无法运行 python 命令 | Windows 环境未安装 Python | 改用 Node.js 方案替代 | 执行前先用 `which python` 或 `where python` 检查环境 |
| 6.3 | npm | `require('docx')` 找不到模块 | Shell cwd 被重置，包安装到错误目录 | `cd /d/AI引路人项目 && npm install docx` | npm install 前先用 `pwd` 确认当前目录 |
| 6.3 | JavaScript | `SyntaxError: Unexpected identifier 'AI'` — 中文引号导致语法错误 | 中文弯引号 `"` `"` 被 JS 解析为字符串结束符 | 改用 Unicode 转义 `\u201C` `\u201D`，或使用模板字符串 | 代码中的中文文本优先用模板字符串（反引号）包裹 |
| 6.3 | Bash | `unexpected EOF while looking for matching '"'` — 嵌套引号崩溃 | 多层引号嵌套（bash 引号 + JS 引号 + HTML 引号）导致解析失败 | 将脚本写入独立 `.js` 文件再执行，避免 `node -e` 嵌套复杂内容 | 复杂脚本不内联到命令行，写独立文件执行 |
| 6.3 | sed | `sed -i` 替换中文引号无效 | sed 在 Windows Git Bash 中对 UTF-8 多字节字符处理不一致 | 改用 Node.js 脚本进行字符替换 | Windows 环境优先用 Node.js 而非 sed 处理文本替换 |
| 6.3 | Windows | `python` 命令被 Windows Store 存根拦截 — 已安装的 Python 3.12 无法通过 `python` 调用 | `C:\Users\...\WindowsApps\python.exe`（Store存根）在 PATH 中优先级高于真实 Python | 删除 WindowsApps 下的 `python.exe`/`python3.exe` 存根 + 将真实 Python 路径写入 `~/.bashrc` | 安装 Python 后先 `which python` 确认路径，若指向 WindowsApps 则立即处理 |
| 6.3 | Agent | 第二阶段考核中，用户输入"来"时，Agent 输出了第一阶段第5题（Prompt设计）而非第二阶段第5题（实操检验） | **阶段锚点丢失**：短输入"来"触发"第5题"检索，知识库未做阶段过滤，错误命中 Phase1 的同名题号 | 1. 考核中维护 `current_phase` 上下文锚点 2. 短输入(≤2字)先回查确认再输出 3. 知识库"第X题"条目带 phase 标签 | 1. 阶段标记持久化 2. 短输入防御协议 3. 知识库检索强制阶段过滤 |

## 每周复盘

### 2026年第23周（6.2-6.8）

- 新错误数：8
- 重复错误数：0（4号错误"阶段锚点丢失"为新发现，未重复）
- 核心教训：
  1. **执行前检查环境** — 假设工具已安装是最大的坑；安装后必须验证 PATH
  2. **中文特殊字符谨慎处理** — 编程环境中用模板字符串或 Unicode 转义
  3. **复杂操作写入文件** — 不要在一行命令里塞所有逻辑；bash 嵌套引号是常见陷阱
  4. **Windows 特有坑** — sed UTF-8 兼容差、Windows Store 存根拦截真实命令
  5. **阶段上下文锚定**（考核专用）— 短输入(≤2字)必须先回溯确认阶段再输出；知识库检索必须带阶段标签过滤
