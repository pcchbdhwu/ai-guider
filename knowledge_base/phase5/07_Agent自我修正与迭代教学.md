# Agent 自我修正与迭代教学

## 核心理念
Agent 和人一样，会在执行任务时犯错。一个成熟的 Agent 体系需要内置**自我修正机制**——能在每次失败后自动记录错误、分析原因、调整策略，避免重复犯错。

## 一、Agent 常见的四类错误

| 错误类型 | 表现 | 示例 |
|---------|------|------|
| **工具调用错误** | 调用不存在的工具、参数格式错误 | 调用 `read_file` 实际工具名是 `Read` |
| **任务理解偏差** | 输出偏离用户意图 | 用户要 JSON 配置，Agent 输出了 YAML |
| **执行顺序错误** | 跳过必要步骤直接执行后续操作 | 未读文件直接尝试 Edit（会失败） |
| **环境假设错误** | 假设了错误的环境条件 | 假设 Python 已安装，实际未安装 |

## 二、Agent 自我修正四步循环

```
执行任务 → 遇到错误 → 错误分析与记录 → 调整策略 → 重新执行
   ↑                                                      |
   └──────────────────────────────────────────────────────┘
```

### Step 1：错误检测
Agent 应当主动检测以下信号：
- 工具调用返回 error/非预期结果
- 命令执行 exit code ≠ 0
- 输出结果与预期格式不匹配
- 用户显式指出错误

### Step 2：错误分析
每次错误发生时，Agent 自动问自己：
1. **发生了什么**（错误信息原文）
2. **为什么发生**（根因分析）
3. **是知识不足还是执行失误**（分类）
4. **下次如何避免**（改进措施）

### Step 3：策略调整
根据错误类型选择修正策略：
- **工具名/参数错误** → 查询可用工具列表，修正参数
- **环境假设错误** → 先检查环境（如 `which python`），再执行
- **格式不匹配** → 重新确认用户要求的格式
- **权限不足** → 向用户说明需要的权限

### Step 4：记录与学习
将错误记录到知识库或记忆系统：
```
错误记录模板：
- 时间：[timestamp]
- 任务：[原始任务描述]
- 错误：[完整错误信息]
- 根因：[为什么出错]
- 修正方案：[怎么解决的]
- 预防措施：[以后怎么避免]
```

## 三、Cherry Studio 配置建议：错误追踪

### 3.1 启用全局记忆
Cherry Studio → 设置 → 全局记忆，添加：
```
我使用的操作系统：Windows 11
已安装的工具：Node.js, Git, npm
常用 Shell：Git Bash
语言偏好：中文
```

### 3.2 在 Agent 系统提示词中加入错误处理规则
```markdown
## 错误处理规则
1. 每次遇到错误时，先不要放弃，尝试分析错误原因
2. 如果是工具调用失败，检查工具名称和参数是否正确
3. 如果是环境问题（如命令未找到），先诊断环境再执行
4. 如果同一错误出现 2 次，改变策略而非重复尝试
5. 每次解决错误后，用一句话总结教训
```

### 3.3 创建专门的"排障 Agent"
在 Cherry Studio 中单独创建一个 Agent：
- **名称**：排障顾问
- **模型**：Claude（推理更强，适合分析错误）
- **系统提示词**：专门分析报错信息、给出排查路径
- **使用时机**：当其他 Agent 反复失败时，切换到排障顾问分析问题

## 四、今日 Word 文档生成的错误案例（实战教学）

### 错误回放
今天在生成实验报告 Word 文档时，经历了以下错误：

**错误1：Python 不可用**
- 尝试：`python -c "from docx import Document..."`
- 错误：`Python was not found`
- 根因：Windows 环境未安装 Python
- 修正：改用 Node.js 方案

**错误2：Node.js docx 包安装位置错误**
- 尝试：在默认目录 `npm install docx`
- 错误：安装后 `require('docx')` 找不到
- 根因：shell cwd 被重置，包安装到了错误目录
- 修正：在 D 盘目标目录重新安装

**错误3：中文引号导致 JS 语法错误**
- 尝试：`"基于 Cherry Studio 智能体的"AI 引路人""`
- 错误：`SyntaxError: Unexpected identifier 'AI'`
- 根因：中文弯引号 `"` `"` 被 JS 解析为字符串结束符
- 修正：改用 Unicode 转义 `\u201C` `\u201D`（sed 修复失败），最终换方案用 HTML 生成 .doc

**错误4：Bash 嵌套引号崩溃**
- 尝试：`node -e "..."` 中内嵌复杂 HTML 字符串
- 错误：`unexpected EOF while looking for matching '"'`
- 根因：多层引号嵌套导致 bash 解析失败
- 修正：将脚本写入独立文件再执行

### 教训总结
| 教训 | 预防措施 |
|------|---------|
| 先检查环境再执行 | 执行前 `which python` `which node` |
| 中文特殊字符注意转义 | JS 中用 `\uXXXX` 或模板字符串 |
| 复杂脚本不要内联 | 写入 .js 文件再执行 |
| npm 安装注意 cwd | 用 `cd /target && npm install` |
| 遇到 2 次同样错误就换方案 | 不固执修补同一方法 |

## 五、给 Agent 的自我修正提示词模板

将此段添加到各阶段 Agent 的系统提示词中：

```markdown
## 自我修正机制
当执行操作失败时，按以下流程处理：

1. **记录错误**：记住完整的错误信息
2. **分类判断**：
   - 工具名称/参数错误 → 查阅可用工具列表后修正
   - 环境缺失 → 先诊断环境（检查命令是否存在）
   - 权限问题 → 向用户说明需要的权限
   - 格式问题 → 重新确认用户期望的格式
3. **最多尝试 2 种不同方案**：如果第 2 种方案也失败，向用户说明情况并请求指导
4. **事后总结**：问题解决后，用一句话记录原因和解决方案
```
