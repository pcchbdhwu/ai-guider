# DeepSeek 前缀缓存优化指南

## 缓存机制本质
DeepSeek 使用**磁盘级上下文缓存**，对所有用户默认启用。核心规则：**字节级精确匹配**——哪怕一个字符不同（如时间戳变化），整个前缀缓存即失效。

**缓存命中折扣**：命中 token 仅 ¥0.025/百万（原价 ¥3/百万），折扣率 98%。

## 六大缓存破坏模式（避坑指南）

| 模式 | 症状 | 防御 |
|------|------|------|
| **A. 动态前缀注入** | System Prompt含时间戳/UUID/路径 | 替换为静态占位符：[TIME]/[ID]/[TEMP] |
| **B. 全量上下文重发** | 每轮拼接完整历史（Cherry Studio等客户端常见） | 使用增量上下文发送 |
| **C. 工具定义变化** | MCP 连接/断开导致工具描述变化 | 工具名字母排序+缓存定义输出 |
| **D. Agent 重建** | 新实例注入当前时间戳 | 跨重建复用静态 system_prompt |
| **E. Lookback 窗口超限** | Anthropic特有：缓存回溯不超20 block | 预先放置第二缓存断点 |
| **F. 跨实例不共享** | 多终端多 user_id 无法共享缓存 | 确定性 user_id（Git Root hash） |

## 五层优化技术栈

| 层 | 代表工具 | 做法 | 收益 |
|----|---------|------|------|
| 1. 字节级前缀固化 | Reasonix | 正则替换动态内容为占位符 | 防灾难+1-4pp提升 |
| 2. 上下文结构重排 | NanoBot | 易变数据从前缀移到后缀 | 前缀跨轮次永久不变 |
| 3. 缓存键策略 | NanoBot/ds4 | cache_key 只用 messages[:2] | 后续追加不影响命中 |
| 4. Provider API 层 | Anthropic/OpenAI | max_tokens:0 预热缓存断点 | 服务启动时预热 |
| 5. 推理引擎层 | vLLM/DeepSeek MLA | FP8 KV 量化+MLA架构 | 内存 135GB→8GB |

## 实战数据
- **Reasonix**：一天 4.35 亿 token，99.82% 命中率，$61→$12（5x成本削减）
- **京东云**：HashTrie 算法，命中率 61%→92%，吞吐 +29.9%
- **反面案例**：Hermes-WebUI 成本是 CLI 的 100x（Agent重建+全量重发）

## 日常使用建议
1. **不要每轮重发完整历史**——只发送新增内容
2. **定期新建会话**——每 10-15 轮建新会话
3. **避免会话中途切换工具配置**
4. **监控 prompt_cache_hit_tokens**——在成本面板中查看命中率
5. **System Prompt 中不要放时间戳/动态路径**
